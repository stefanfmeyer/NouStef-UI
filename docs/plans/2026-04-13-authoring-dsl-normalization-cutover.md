# Authoring DSL Normalization Cutover

Date: 2026-04-13

## Goal

Replace the current production Hermes Home enrichment path with a stricter baseline-first, authoring-DSL-first architecture:

1. Baseline Home workspace always succeeds first.
2. Hermes generates a smaller authoring DSL, not TSX and not the strict persisted workspace model.
3. The bridge injects a compact schema-reference packet and examples into the DSL prompt.
4. The bridge validates, auto-repairs, and normalizes the authoring DSL locally.
5. The bridge persists only the normalized `workspace_model` and `workspace_patch`.
6. The browser renders only from the normalized local model.
7. No standard production flow enters runtime TSX applet generation, manifest synthesis, verification, or promotion.

## Two Schema Layers

### 1. Authoring DSL generated by Hermes

New production contract:

- `kind: "workspace_dsl"`
- `schemaVersion: "space_workspace_dsl/v2"`
- `sdkVersion: "workspace_sdk/v1"`

Top-level shape:

- `title`
- `summary`
- `subtitle`
- `status`
- `tabs`
  - semantic tabs with `id`, `label`, `sectionIds`, `layout`
- `datasets`
  - bridge-facing data references with `id`, `title`, `display`, `fields`, optional notes, and optional focus entity hints
- `sections`
  - higher-level section descriptions such as summary, dataset, comparison, timeline, detail, notes, links, callout, actions, or status
- `actions`
  - declarative action references limited to allowed action ids and `existing_action`
- `operations`
  - high-level update intents against datasets, tabs, sections, notes, action state, selection, or workspace status
- `semanticHints`
  - preferred layout
  - narrow-pane strategy
  - primary dataset
  - comparison focus
  - display priority
- `metadata`
  - optional low-risk hints only

Rules:

- no TSX
- no final `workspace_model`
- no final `workspace_patch`
- no direct low-level `WorkspaceEntity`, `WorkspaceCollection`, `WorkspaceView`, or `WorkspaceSection` authoring as the primary contract
- no undeclared action handlers
- no arbitrary extra top-level keys

### 2. Strict internal normalized model owned by the bridge

Persisted production output remains:

- `workspace_model`
  - `kind: "workspace_model"`
  - `schemaVersion: "space_workspace_model/v1"`
  - `sdkVersion: "workspace_sdk/v1"`
  - `revision`
  - `title`
  - `subtitle`
  - `description`
  - `status`
  - `entities`
  - `collections`
  - `views`
  - `sections`
  - `tabs`
  - `actions`
  - `state`
  - `metadata`
- `workspace_patch`
  - `kind: "workspace_patch"`
  - `schemaVersion: "space_workspace_patch/v1"`
  - `sdkVersion: "workspace_sdk/v1"`
  - `baseRevision`
  - `nextRevision`
  - `summary`
  - deterministic patch operations

The bridge, not Hermes, owns this translation.

## Prompt Contract Injection

For every DSL-generation and DSL-repair prompt, inject a compact schema-reference packet built locally from protocol metadata plus the current normalized workspace context.

Required packet contents:

1. concise contract summary
2. exact allowed top-level keys
3. exact allowed section kinds
4. exact allowed action kinds
5. exact allowed operation names
6. required fields
7. one or two valid examples
8. one or two invalid examples with a one-line reason each
9. explicit statement that unknown keys will be dropped and ambiguity will fail

The packet must stay digestible. Do not dump raw Zod/schema blobs.

## Normalization Pipeline

### Baseline lane

1. Run Hermes task.
2. Persist transcript-visible answer.
3. Generate or recover the minimal `hermes_space_seed/v1`.
4. Build the baseline Home workspace locally.
5. Persist baseline `workspace_model` and `workspace_patch`.
6. Complete the user-facing request.

### Async enrichment lane

1. Start `dsl_enrichment` build.
2. Generate `workspace_dsl`.
3. Extract the first balanced JSON object.
4. Validate against the authoring DSL schema.
5. Run deterministic local auto-repair.
6. Re-validate the repaired authoring DSL.
7. Normalize authoring DSL into strict `workspace_model`.
8. Synthesize deterministic `workspace_patch`.
9. Persist `workspace_dsl`, `workspace_model`, and `workspace_patch`.
10. Promote only the normalized model.

No standard production branch may continue into TSX generation.

## Local Auto-Repair Rules

Apply deterministic repair before any Hermes repair call:

- drop unrecognized top-level keys
- drop unrecognized section/action/operation keys
- map common aliases
  - `description` -> `summary` when `summary` missing
  - `views` -> `sections` when shape clearly matches
  - `collections` -> `datasets` when the authored shape is high-level
  - `primaryDatasetId` -> `semanticHints.primaryDatasetId`
- map common operation synonyms
  - `set_tab` -> `update_tab`
  - `set_section` -> `replace_section`
  - `upsert_section` -> `replace_section`
  - `set_notes` -> `append_notes`
- normalize action shorthand
  - string action id -> `{ id, kind: "prompt" | "bridge" }` only when the id is already allowed by the baseline action registry
- fill safe defaults
  - derive `workspace.summary` from assistant context when missing
  - derive `workspace.title` from baseline title when missing
  - derive `primaryTabId` from first valid tab
- coerce safe scalar variants
  - singleton object -> one-element array where the field requires an array
  - numeric strings -> numbers where the schema explicitly allows numbers
- filter dangling references
  - sections referencing unknown datasets
  - tabs referencing unknown views
  - actions not present in the allowed baseline action set

If repair is ambiguous, fail instead of guessing.

## Hermes Repair Loop Design

Bounded repair only. No loops.

### Trigger

Run the repair pass only if:

- authoring DSL extraction succeeded but schema validation failed, or
- local auto-repair still leaves a schema-invalid or normalization-invalid authoring DSL

### Prompt contents

The repair prompt may include only:

- the invalid DSL
- the exact validation errors
- the narrow relevant contract excerpt
- instruction to return JSON only

The repair prompt must not include:

- original task prompt
- new live task instructions
- tool instructions
- search/email/rerun directions
- external context fetch requests

### Budget

- one repair pass by default
- a second pass only if the first repaired output is parseable JSON but fails on a single narrow contract issue that is demonstrably repairable without ambiguity
- otherwise stop and preserve the baseline

### Persistence

Repair always uses persisted artifacts only:

- baseline `workspace_model`
- baseline `workspace_patch`
- `user_prompt`
- `intent`
- `raw_data`
- `assistant_context`
- `analysis`
- `normalized_data`
- prior `workspace_dsl` if present
